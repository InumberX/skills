# 書き方の原則

## 段階的読み込み(progressive disclosure)

スキルは 3 段階で読み込まれる。各段階の予算を意識して書く:

1. **frontmatter(name + description)** — 常にコンテキストに載る。発動判断はここだけで行われる
2. **SKILL.md 本文** — 発動時に読み込まれる。目安 150 行以内。手順と「どのファイルをいつ読むか」の案内に徹する
3. **rules/ examples/ などの分割ファイル** — 必要なときだけ読み込まれる。詳細はすべてここに置く

SKILL.md には、分割ファイルを「いつ読むか」のマッピング表を必ず付ける。案内がない分割ファイルは読まれないか、逆に毎回全部読まれるかのどちらかになる。

```markdown
| 依頼パターン | 参照ファイル |
|---|---|
| 「全体レビュー」 | `rules/**/*.md` 全て |
| 「命名だけ」 | `rules/style/naming.md` |
```

## 機械検出できるルールは書かない

lint・formatter・typecheck で検出できる項目(フォーマット、未使用変数、import 順序など)はスキルに書かない。ツールの方が確実で、スキルは**人間の判断が必要なルール**に集中させた方が読み込みコストに見合う。書く場合は「`npm run lint` で検出可能」と明記して例外扱いにする。

## 理由(why)を書く

ルールには「何をするか」だけでなく「なぜそうするか」を添える。理由があると、ルールに書かれていない周辺ケースでも意図に沿った判断ができる。理由を書けないルールは、そもそもルールにすべきか疑う。

- Bad: 「Element には `_` を 1 つ使うこと」
- Good: 「Element は `_` 1 つ、Modifier は `__` 2 つ。区別がつかないと修飾子の組み合わせ指定が壊れるため」

同様に、`MUST`・`NEVER` の連打や過度に硬い構造は避け、命令形 + 理由で書く。

## 例は good / bad をセットで

規約の例示は「良い例」と「悪い例」を対で示す。悪い例には**何が悪いか**のコメントを付ける。判断に迷うケースこそ例にする価値がある(明白なケースは例がなくても判断できる)。

## 特定の一例に過剰適合させない

スキルは今後さまざまな依頼に対して使われる。作成のきっかけになった具体例だけで通用する記述(その場限りのファイル名・数値・手順)になっていないか、書き終えたら見直す。具体例は examples/ に置き、SKILL.md と rules/ には一般化した原則を書く。

## 成長のさせ方

- 新しい観点を足すときは、まず既存ファイルへの追記で済まないか検討する。ファイルを増やすのは、読み込み単位を分けたい(常に一緒には読まれたくない)ときだけ
- 分割ファイルを追加したら、SKILL.md のマッピング表にも必ず行を追加する
- SKILL.md が 150 行を超え始めたら、詳細を rules/ へ追い出して入口の薄さを保つ
