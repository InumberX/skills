# 配置の線引き

スキル・ルールを書く場所は 3 つある。それぞれ役割が違うため、内容ではなく**性質**で置き場所を決める。

| 置き場所 | 役割 | 読み込まれ方 |
|---|---|---|
| 各リポジトリの `CLAUDE.md` | プロジェクトの地図(構成・コマンド・技術選定) | 常時 |
| 各リポジトリの `.claude/skills/` | プロジェクト固有のタスク手順・規約 | タスク発動時 |
| 本リポジトリ(`skills/`) | プロジェクト横断のタスク手順・規約 | 取り込み先で発動時 |

## 判断基準

上から順に確認し、最初に該当したところに置く:

1. **実コードのパス・識別子を引用するか?**(例: `BaseButton/`、`app/styles/variables/`)
   → 各リポジトリの `.claude/skills/` に置く。コードと同じ PR で更新される場所に置かないと、コード変更時に更新されず腐る。
2. **常に知っておくべき前提情報か?**(手順ではなく地図)
   → `CLAUDE.md` に置く。スキルにすると必要なときに読まれない可能性がある。
3. **複数プロジェクトで同じ内容になるか?**
   → 本リポジトリに置く。例: commit message 規約、GitHub Flow、レビュー手順の骨格。
4. **迷ったら** → 各プロジェクト側に置く。中央への昇格は後からできるが、中央に置いたものが特定プロジェクトに引きずられると全プロジェクトに影響する。

## プロジェクト → 中央への昇格

同じ内容のスキル(または骨格)が **2 つ以上のリポジトリにコピーされた時点**で昇格を検討する。昇格時は:

1. プロジェクト固有部分(コードパス引用・技術スタック依存の記述)と共通部分を分離する
2. 共通部分だけを本リポジトリに移し、プロジェクト側には固有部分を残す
3. 実例: `review-pr` は SKILL.md の骨格(手順・出力フォーマット・拡張ガイド)が after_works-v006 と sugidama でほぼ同一だが、`rules/` の中身(CSS Modules の PascalCase vs Vanilla Extract の camelCase BEM)は完全に別物。昇格するなら骨格のみ

## 取り込み方法(当面)

本リポジトリのスキルを各プロジェクトで使う場合は、当面は手動コピーで取り込む。取り込み先で改善した場合は本リポジトリに還元する。submodule や同期 CI は、重複更新が実際に苦痛になってから導入する。
